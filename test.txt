In computer networking, reliable data transfer over an unreliable channel is a
fundamental challenge. Protocols such as TCP solve this problem by using
mechanisms like acknowledgements, sequence numbers, retransmissions, and
sliding window flow control. However, in this project we simulate similar
behavior using UDP, which by default provides no reliability guarantees.

The goal of this implementation is to build a simplified reliable file transfer
protocol on top of UDP sockets. The client divides a file into multiple fixed
size chunks. Each chunk is assigned a sequence number and transmitted to the
server as a packet containing metadata such as file identifier, sequence
number, total number of chunks, and the chunk data itself encoded in Base64.

Because UDP does not guarantee packet delivery, order, or duplication
protection, the protocol must handle these conditions manually. The client
implements a sliding window mechanism that allows multiple packets to be sent
before waiting for acknowledgements. This improves throughput while still
maintaining reliability. The server responds to each successfully received
chunk with an acknowledgement containing the sequence number.

To demonstrate reliability features, the server intentionally simulates packet
loss by ignoring the first occurrence of every chunk. When the client does not
receive acknowledgements for certain packets within a timeout period, it
retransmits the missing chunks. This behavior mimics the retransmission logic
found in real-world reliable transport protocols.

Another important feature implemented in this protocol is transfer resumption.
If the client process stops before the entire file has been transmitted, it can
restart and query the server for which chunks were already received. The server
responds with a list of stored sequence numbers. The client then resumes
transmission by sending only the missing chunks instead of starting the entire
transfer again. This prevents redundant data transmission and improves
efficiency for interrupted transfers.

Once all chunks have been successfully received, the server reconstructs the
original file by concatenating the chunks in order of their sequence numbers.
The reconstructed output file should be identical to the original input file
sent by the client. This confirms that the reliability mechanisms worked
correctly despite packet loss and possible interruptions.

This project demonstrates important networking concepts including UDP socket
communication, custom packet formats using JSON, Base64 encoding for binary
data transmission, sliding window protocols for flow control, acknowledgement
based reliability, retransmission strategies, and resumable file transfers.
Such mechanisms are foundational concepts used in many real-world distributed
systems and networking protocols.

If the client process stops before the entire file has been transmitted, it can
restart and query the server for which chunks were already received. The server
responds with a list of stored sequence numbers. The client then resumes
transmission by sending only the missing chunks instead of starting the entire
transfer again. This prevents redundant data transmission and improves
efficiency for interrupted transfers.

Once all chunks have been successfully received, the server reconstructs the
original file by concatenating the chunks in order of their sequence numbers.
The reconstructed output file should be identical to the original input file
sent by the client. This confirms that the reliability mechanisms worked
correctly despite packet loss and possible interruptions.

In computer networking, reliable data transfer over an unreliable channel is a
fundamental challenge. Protocols such as TCP solve this problem by using
mechanisms like acknowledgements, sequence numbers, retransmissions, and
sliding window flow control. However, in this project we simulate similar
behavior using UDP, which by default provides no reliability guarantees.

The goal of this implementation is to build a simplified reliable file transfer
protocol on top of UDP sockets. The client divides a file into multiple fixed
size chunks. Each chunk is assigned a sequence number and transmitted to the
server as a packet containing metadata such as file identifier, sequence
number, total number of chunks, and the chunk data itself encoded in Base64.

Because UDP does not guarantee packet delivery, order, or duplication
protection, the protocol must handle these conditions manually. The client
implements a sliding window mechanism that allows multiple packets to be sent
before waiting for acknowledgements. This improves throughput while still
maintaining reliability. The server responds to each successfully received
chunk with an acknowledgement containing the sequence number.

To demonstrate reliability features, the server intentionally simulates packet
loss by ignoring the first occurrence of every chunk. When the client does not
receive acknowledgements for certain packets within a timeout period, it
retransmits the missing chunks. This behavior mimics the retransmission logic
found in real-world reliable transport protocols.
